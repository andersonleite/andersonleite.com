Esse post relata minha ida a <a href="http://www.qconsf.com">QCon SF</a>, um evento repleto de personalidades influentes sobre desenvolvimento de software como martin Fowler, Ola Bini, Neal Ford, Charles Nutter, Tomas Enebo, Dan North, entre outros(http://qconsf.com/sf2009/speakers/)

O evento não é sobre java, não é sobre ruby. O evento é sobre software, e uma série de tecnologias estão sendo comentadas por aqui. Hoje por exemplo tivemos tracks sobre Kanban/Lean, Architecture, Browser como platforma, Java, Ruby , performance e escalabilidade.

O mais curioso foi a aparição exatamente da track sobre Ruby. Uma sala voltada apenas para palestras sobre a linguagem foi montada o que mostra a força que a linguagem vem tomando.
Foram 5 palestras sobre o assunto mais um keynote introdutório do Charles Nutter (JRuby).
Charles Nutter abriu a track comentando dos quase 5 anos do projeto JRuby, comentando que muita coisa vem mudando
nesse tempo e apresentando os palestrantese.
A primeira palestra foi de ninguém menos que yukihiro matsumoto, criador da linguagem ruby.
Matz iniciou a palestra com o título "Why we love Ruby" explicando que existem dois tipos de rubistas: os que amam ruby e os que não. Alguns utilizam ruby apenas por causa do Rails, outros realmente adimiram a linguagem.
Antigamente os motivos para gostar de Ruby estam no fato de ser divertido, focado em humanos, produtividade e eficiência. Hoje em dia existe uma nova filosofia denominada <b>QWAN</b>. QWAN significa <b>Quality Without A Name</b>, uma filosofia baseada em
princípios como participação, crescimento, padrões, analises e coordenação.
Esses princípios são muito próximos do Agile, motivo pelo qual Ruby e Rails estão tão próximos dessa metodologia de desenvolvimento.
Matz ainda disse que se pudesse refazer o Ruby ou criar uma nova linguagem ela seria mais funcional, reforçando a influencia que lingugens funcionais tem causado no desenvolvimento de software atualmente.
oas se sentirem mais confortaveis

O próximo palestrante foi Micah martin com o título <b>Basking in the limelight</b>.
limelight(http://limelight.8thlight.com/main/sparkle) é um projeto escritpo em ruby para criar aplicações desktop, com muitas idéias da web, bem baseado no modo Rails de estruturar o projeto. O projeto depende do jruby.
Continuando tivemos a apresentação do Tomas Enebo, arquiteto do projeto JRuby, mostrando as vantagens de se utilizar
ruby sobre a plataforma java, status do projeto, pontos positivos e negativos.
Atualmente JRuby é 100% compatível com ruby 1.8.7 e existe um mode para rodar com ruby 1.9 quase completo.
Compatibilidade é um dos pontos importantes do projeto que conta com 37000 testes passando, 22000 assertions.
Alguns dos problemas citados foram questões de incompaatibilidade, como missing some posix behavior (no fork()), no continuations (callcc), slower startup e cannot run native C extensions(o maior deles atualmente).
Tomas Enebo mostrou também os pontos positivos mostrando a otimização realizada pela JVM, hotspots, debbugers paralelismo e garbage collector.
A conclusão é que a jvm é hoje uma excelente base para um projeto ruby, e que muitas bibliotecas podem se aproveitar desse fato.
O próximo palestrante foi yehuda kartz que trabalha na implementação do Rails3 que será uma fusão entre o Merb e o Rails atual.
Merb é um projeto com 3 anos, focado em solucionar alguns problemas encontrados no Rails. (exemplo: ao fazer um upload em Rails tudo para até que o upload complete, Rails não é thread-safe).
Uma das novas mudanças estão na questão das dependencias. Atualmente não é possível usar um "layout" sem declarar toda sua hierarquia. No Rails 3 todas as dependecias terao sido "mergiadas", para evitar esse tipo de problema.
Outro ponto importante está no fato do Rails ser modificado para facilitar o suporte a DataMapper ou Sequel como ORM principal.
Será possível configurar suas opções fazendo algo como:
config.generators do |g|
g.orm :datamapper
g.template_engine :haml
g.test_engine :rspec
end
Além disso a declaração de gems por ambiente será adicionada, permitindo algo como:
only [:test, ; development] do |gem|
gem.config :rspec
# outras gems
end
Modificações nas rotas ocorrerão também porém a compatibilidade com as rotas atuais será mantida.
Em resumos, as principais novidades do Rails 3 são : mais modular, performatico, agnostico em relação a ORM, melhor utilizaço do Rack e funcionalidades do Merb.
A última apresentação foi do Neal Ford, contando sua experiência em um dos maiores projetos utilizando Ruby on Rails no mundo, o OVE.com.
O projeto contou com 11 pares programando, XServe(selenium grid), 4 diferentes ambientes , incluindo QA
10 web boxes, 2 images servers, 4 databases servers, backgorud server e memcached
Algumas das lições aprendidas pela equipe foram:
-tecnology isnt as important as responsiveness to business needs
- dont try to convince too early
- demonstration over arguments
- scale infrastructure oportunistically .. but dont wait too long
-fight the battle to keep tests fast
-invent stuff if you have to
-write smart tests
-scale dev just like prod
- software is more about communication than tecnology
